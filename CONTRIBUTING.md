# Contributing

Thanks for your interest. Baseplate is a personal starter — happy to take
PRs that fit the scope, but not actively recruiting maintainers.

## Setup

See [README.md](README.md#quick-start) for the full version. Short:

```bash
cp .env.example backend/.env
cp .env.example frontend/.env.local
make install
make install-hooks   # registers pre-commit hooks
make hash-password   # paste the output into backend/.env as ADMIN_PASSWORD_HASH
make db              # start Postgres in Docker
make migrate
make dev
```

`CLAUDE.md` and `.github/copilot-instructions.md` document the conventions
agents (and humans) should follow when extending the codebase.

## Branch and commit conventions

- Branch off `main` with a short descriptive name (e.g. `add-search`,
  `fix-csrf-cookie-bug`).
- Commit messages are short imperative ("Add X", "Fix Y"). Multi-topic
  commits use semicolon separators ("Add X; refactor Y"). Wrap bodies at
  ~72 cols.
- One topic per PR. Multiple commits per PR is fine if each tells a story.

## Tests

```bash
make lint           # ruff + tsc + ESLint
make test-backend   # pytest — uses SQLite via aiosqlite, no Postgres needed
make test-frontend  # vitest + happy-dom
```

For new features:

- Backend services: unit-style tests against the service class.
- API endpoints: integration tests using the `client` fixture in
  `backend/tests/conftest.py`. The `_login()` helper in `test_items.py` is
  the canonical pattern (returns the CSRF token; pass it as
  `headers={"X-CSRF-Token": csrf}` on writes).
- React components: add a test in `frontend/src/__tests__/` following the
  `Button.test.tsx` pattern.

## Before opening a PR

The pre-commit hooks catch most of this automatically, but:

- `make lint` clean
- `make test-backend` and `make test-frontend` pass
- New migrations look right (read the autogenerated SQL before accepting)
- UI changes verified in the browser (type-checks aren't enough)

CI runs all the above on every PR.

## Architecture changes

If you're proposing something cross-cutting (new middleware, auth flow
changes, schema migrations that touch existing tables), open an issue
first to discuss the approach. Saves wasted PR work.

## Code style

Style is enforced by ruff (Python) and ESLint (TypeScript). Don't fight
the linter. If you have a strong opinion about a rule, propose the
config change in a separate PR.
